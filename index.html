大模型Token行业背景

大模型驱动的新软件范式：软件的核心正在从“人操作软件”演变为“人指挥 AI，AI 操作软件”。

大模型推动软件范式迁移

过去 20 年软件演进路径，当前正处于“辅助智能化”向“自主执行化”的关键跃迁阶段。

Stage 1
SaaS
软件以标准化功能和固定流程为核心，人通过界面操作系统完成查询、录入、审批和执行。
Stage 2
Copilot
AI 开始参与内容生成、搜索总结和辅助决策，显著提升人的效率，但执行主导权仍掌握在人手里。
Stage 3
Agent
AI 可以理解目标、调用工具、跨步骤执行任务，并在流程中根据环境反馈动态调整策略和输出。
Stage 4
Harness
围绕 AI Agent 尤其是 Coding Agent，构建约束机制、反馈回路、工作流控制和持续改进循环的系统工程实践。

企业面临的核心挑战

尽管 AI 能力强大，但落地存在断层，企业仍面临“用不起来”的困境。

落地断层正在从“调用模型”转向“运营系统”

企业不是缺少模型，而是缺少一套能够承接多模型接入、业务编排、权限治理和持续运行的统一网关。

工具碎片化

  • 大量系统如 CRM、BI、ERP、Notion 与各类模型服务并存，接口、权限和数据格式彼此割裂。
  • 数据与动作分散在多个系统里，AI 很难通过统一入口稳定调用和跨系统编排。
02 Execution Gap

AI 无法真正执行

  • 多数 AI 产品仍停留在对话和回答层，缺少与真实业务工具、流程和权限体系的可靠接驳。
  • 没有统一网关和工具编排能力，AI 难以把理解、决策和执行串成完整闭环。
03 Low Automation

自动化程度低

  • 关键流程依然依赖人工驱动和人工兜底，AI 难以跨系统持续运行、感知状态并自我修正。
  • 缺少长效路由、反馈与治理机制后，企业很难把 AI 从试点工具升级为稳定生产能力。
大模型正把软件从“工具时代”推进到“智能执行时代”，企业迫切需要全新的 AI 网关与入口来承载并管理这一变革。

Token 介绍

Token 是模型处理文本时的最小单位。理解 token，才能理解上下文长度、响应成本和输入输出的方式。

什么是 Token

可以把 token 理解成模型切分文本时使用的“文本片段”。不同语言、标点和空格的切分方式不同，因此 token 数不等于字数或词数。

English
1 word ≈ 1 token
英文通常按词或词片段切分，一个单词常常接近 1 个 token。
中文
1 字 ≈ 1-2 token
中文常常按字或子词切分，单个汉字可能拆成 1 到 2 个 token。
标点与空格
也算 token
标点、空格和部分特殊符号都会被单独计入 token 数。
01 Context Window

决定模型一次能“看见”多少内容

输入越长，消耗的 token 越多；超过上下文窗口后，模型就无法完整记住前面的信息。

02 Cost Metering

决定调用成本如何统计

很多平台按输入 token + 输出 token 计费，所以同样一句话，语言和写法不同，成本也会不同。

计算示例

下面的示例只用于帮助建立直觉，实际 token 数会因模型分词器不同而略有差异。

A English Example

"Hello, world!" ≈ 3 tokens

  • Hello = 1 token
  • , = 1 token
  • world = 1 token
  • ! 可能与前面的内容合并，具体取决于分词器
B Chinese Example

"你好，世界！" ≈ 6-8 tokens

  • 中文字符通常会被切成 1-2 个 token
  • 标点符号也会单独计入 token
  • 句子越长，token 数增长越快
理解 token，是后面理解 API Key、Provider 路由、配额和成本分析的基础。

路由使用场景

Router的真正价值，不是把请求转发出去，而是把效果、成本、稳定性和隔离性沉淀成一套可持续演进的流量策略。

企业最终都会用到路由能力

当模型供应商、账号、环境和业务线同时增加后，流量已经不是“转发逻辑”，而是一层经营能力。

统一入口 多模型接入不改业务代码
灰度升级 新模型上线可观测、可回滚
环境隔离 开发、测试、生产边界清晰
成本与稳定 高性价比运行并具备兜底能力
01 Multi-Model Dispatch

多模型统一入口

业务系统只对接一个 Endpoint，底层由 Router 自动分发到 OpenAI、Anthropic、DeepSeek 或私有模型集群。

  • 前台应用与 Agent 不需要感知供应商差异
  • 不同模型家族走不同 Provider，接入逻辑平台化收口
适合：模型种类多、接入系统多的企业
02 Canary & A/B Test

灰度发布与 A/B 测试

把 5% 到 10% 的流量导向新模型、新 Prompt 或新渠道，在真实业务流量下验证效果、延迟和成本。

  • 支持按 Header、用户组、API Key、Workspace 切流
  • 异常可快速回滚，避免一次性全量切换风险
适合：模型升级频繁、需稳态演进的团队
03 Tenant Isolation

多环境与租户隔离

开发、测试、生产使用不同 Provider 组合和配额策略，不同业务空间也可绑定独立路由边界。

  • 测试流量不污染生产链路，也不干扰生产账单
  • 便于按业务线、客户、项目做资源和权限拆分
适合：多部门、多项目、多租户运营场景
04 Cost & Resilience

成本优化与高可用兜底

常规请求优先走高性价比模型，复杂任务走高阶模型；当主链路异常时自动回退到备用渠道。

  • 把模型调用从“只追求最好”变成“按场景选择最优”
  • 在峰值、限流和故障时维持整体可用性
适合：生产高并发、成本敏感或强 SLA 场景
企业把流量策略从代码细节提升为平台能力，从而持续平衡模型效果、交付稳定性和运营成本。

用户组管理

用户组承接“谁在使用模型能力”，把多个 API Key、权限和计量收口到同一个治理对象，再按环境、团队、项目等边界拆分管理。

用户组承接治理流程

先定义使用主体，再收口 Key、权限和计量，最后形成清晰的责任边界。

Step 1

识别使用主体

先把团队、项目、系统、环境和客户这类调用主体识别出来，明确到底是谁在发起请求、谁应该被统一纳入治理范围。

Consumer

统一组织、授权、归因

  • 把同一主体下散落的多个 API Key 收口到同一个用户组
  • 按用户组统一配置 Router / 模型访问范围，减少逐 Key 配置
  • 按主体独立统计 Token、调用量、成本和审计记录
Step 3

形成责任边界

最终把权限、Token、成本和异常行为都落到同一个责任单元里，管理者可以直接按用户组查看、追踪和调整治理策略。

计费管理

把 Token 消耗、余额扣减、充值补给和账单记录收拢到同一套积分体系里，让企业能清晰地看见“花了多少、还剩多少、为什么花”。

计费链路

一次调用如何从 Token 变成费用，再落到余额和账单。

Step 1

API 调用

请求进入网关后，系统先识别调用主体、模型与上下文长度，再进入计费计算链路。

Step 2

Token 计量

输入 Token 和输出 Token 分别计价，系统按模型单价计算基础费用。

费用 = 输入 Token × 输入单价 + 输出 Token × 输出单价
Step 3

积分扣减

实际费用会根据折扣率计算，最终从账户积分余额中扣减。

计费公式

内部将计费算法和费用示例单独展示，方便理解计量规则和最终扣减结果。

01 Algorithm

内部计费算法

平台先按输入和输出分别统计 Token，再根据对应单价计算基础费用，最后叠加折扣率得到实际扣减金额。

基础费用 = 输入 Token × 输入单价 + 输出 Token × 输出单价 实际费用 = 基础费用 × 折扣率
02 Example

费用示例

输入 Token 1500 × 0.03 / 1000 = 0.045 积分
输出 Token 2000 × 0.06 / 1000 = 0.12 积分
总费用 0.045 + 0.12 = 0.165 积分

折扣配置

折扣能力按“默认策略 → 租户策略 → 精细规则”逐层收口，既能统一报价，又能支持商务分层运营。

全局折扣 默认价格策略

适用于所有用户和所有请求，是平台默认生效的基础折扣层，用于锁定统一报价口径。

租户折扣 按客户或组织定价

针对重点客户、团队或项目设置差异折扣，适合商务谈判、续约激励和客户分层运营。

更具体规则 优先级最高

当模型、路由、租户和 API Key 规则同时命中时，优先使用最具体的配置，避免策略冲突。

Case Focus

统一纳管公有云模型与本地私有模型

客户的诉求不是“多接几个模型”，而是在制造业内网环境里，把不同来源、不同账号、不同路由规则的模型能力收敛成一个可控、可审计、可持续运营的统一入口。

01 Pain Point

核心问题

  • 多个模型品牌与多个云账号并存，入口分散、配置复杂，难以统一管理。
  • 除公有云模型外，还要接入自建 GPU 集群上的私有模型，接入链路更长、运维复杂度更高。
  • 各业务系统各自管理 API Key、路由和访问策略，权限、审计与安全边界容易失控。
02 Solution

方案落点

  • 统一纳管企业自有模型账号、同品牌多云账号和本地私有模型，形成一体化 Provider 管理。
  • 通过 Router 编排不同 Provider、模型与账号，支持负载均衡、自动回退和备用账号切换。
  • 在内网环境里统一配置 API Key、用户组、IP 白名单、内容安全与审计追踪。
03 Value

业务价值

  • 把多入口、多账号、多模型体系收敛成一个统一治理入口，总部管控能力显著提升。
  • 通过路由调度与自动回退机制，降低限流或异常对生产连续性的影响。
  • 让模型资源管理从各系统重复接入，升级为平台化、标准化的长期运营能力。
治理收口 统一 API 入口

业务系统不再直接管理上游模型账号和 Key。

稳定性 多路由兜底

生产峰值与上游异常场景下仍能维持调用连续性。

安全性 内网审计闭环

权限、内容安全与调用追踪全部沉淀在企业内网。
