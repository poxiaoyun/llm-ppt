## 用户组管理

Consumer（用户组）是平台中承接“谁在使用模型能力”的核心对象。
它不是单个 API Key，而是一组调用主体的治理容器，可以把团队、项目、系统或环境统一收口到同一条管理链路上。

一个 Consumer 通常承担三件事：

- 承接多个 API Key，把分散的调用凭证归到同一管理对象下
- 绑定 Router 授权范围，明确这一组使用者可以访问哪些模型能力
- 独立沉淀调用量、Token、成本和审计数据，形成清晰的归因口径

简单理解，Workspace 决定“在哪个业务边界里用”，Consumer 决定“是谁在这个边界里用”。

### Consumer 的管理价值

#### 1. 统一组织 Key
平台中的 API Key 往往会随着应用数量、环境数量和开发协作快速增长。
如果直接以 Key 为单位管理，很快就会失去清晰结构。

通过 Consumer，可以把同一团队、项目或系统的多个 Key 统一归档：

- 一个项目下的开发、测试、生产 Key 可以归到同一个 Consumer
- 一个团队下的不同服务可以共享同一组权限边界
- 管理者可以直接从用户组视角查看资源、权限和消耗情况

#### 2. 形成授权边界
Consumer 是权限控制的重要承接层。
平台不是把每个 Key 单独散配给各种 Router，而是先定义“这一组人或系统能访问什么”，再在组内分发 Key。

这样做的结果是：

- 权限管理从“按 Key 碎片化配置”升级为“按用户组统一授权”
- 可以针对不同团队、项目、客户配置不同模型和路由访问范围
- 新增或替换 Key 时，不需要反复重做底层授权关系

#### 3. 做好使用隔离
Consumer 也是成本分析和运营归因的天然单位。
当平台把调用数据沉淀到用户组层时，企业就能回答很多关键问题：

- 哪个团队调用最多
- 哪个项目成本最高
- 哪个环境存在异常消耗
- 哪组用户应该扩容、限流或调整模型策略

这让成本治理从“看总账”变成“看责任主体”。

### Consumer 和 API Key 的关系

两者不是替代关系，而是分层关系：

- Consumer 负责组织、授权和归因
- API Key 负责具体发起请求

也就是说，API Key 是调用入口，Consumer 是管理单元。
企业真正做治理时，通常不会直接围绕单个 Key 设计体系，而是先定义 Consumer，再在其下发放多个 Key。

### 一句话总结
Consumer 让平台从“管理一堆零散 Key”升级为“管理一组明确的使用主体”，是权限控制、成本归因和协作治理的关键中间层。
