## Router使用场景

Router 的价值不只是“把请求转发出去”，而是把不同业务目标沉淀成一套可复用的流量策略。
对企业来说，Router 往往出现在下面四类高频场景里。

### 场景 1：多模型统一入口
业务系统只对接一个统一 Endpoint，Router 根据模型名、请求路径或 Header 自动分发到合适的 Provider。

- GPT 系列请求走 OpenAI 或兼容渠道
- Claude 系列请求走 Anthropic 渠道
- Qwen、DeepSeek、私有模型走各自专属 Provider
- 前端和业务方不需要感知底层供应商差异

适用价值：降低接入复杂度，让多模型接入从“每个系统各接一遍”变成“平台统一接一次”。

### 场景 2：灰度发布与 A/B 测试
当企业希望验证新模型、新 Prompt 或新上游渠道效果时，可以通过 Router 做流量切分。

- 按 Header、用户组、API Key 或 Workspace 分流
- 90% 流量继续走稳定版本，10% 流量导向新版本
- 对比不同模型在效果、时延、成本上的表现
- 出现异常时一键切回主链路

适用价值：把模型升级从“全量替换”变成“可观测、可回滚的灰度试验”。

### 场景 3：多环境与租户隔离
同一家公司通常同时存在开发、测试、生产环境，以及不同事业部、项目组、客户租户。
这些流量虽然入口统一，但不应该共用同一套上游和配额。

- 开发环境路由到测试 Provider 或低成本模型
- 生产环境路由到正式 Provider 与高可用链路
- 不同业务空间绑定不同 Router 策略和访问范围
- 避免测试流量影响生产稳定性和账单口径

适用价值：实现环境隔离、成本隔离和权限隔离，便于组织化治理。

### 场景 4：成本优化与高可用兜底
并不是所有请求都值得使用最贵、最高规格的模型。Router 可以结合任务类型、模型价格、时延和错误率动态选择最优路径。

- 常规问答优先走高性价比模型
- 复杂推理、代码生成走高阶模型
- 主渠道限流、超时或报错时自动切换备用链路
- 在峰值时段把流量分摊到多个 Provider

适用价值：在保证体验的前提下优化单位请求成本，同时提升整体可用性。

### 一句话总结
Router 让企业可以围绕“效果、成本、稳定性、隔离性”四个目标，持续演进自己的模型调用策略，而不是把流量写死在代码里。
